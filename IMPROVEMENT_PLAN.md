# План улучшений

Этот документ фиксирует практический план развития репозитория `hl7-converter`. План разбит по горизонту выполнения, чтобы можно было отдельно вести быстрые улучшения, структурную доработку и долгосрочное развитие проекта.

## Цели

- Повысить доверие к репозиторию как к поддерживаемому Go-пакету.
- Сделать релизы предсказуемыми и видимыми через GitHub Releases.
- Упростить понимание проекта только по README, без чтения кода.
- Убрать из корня репозитория тестовые и демонстрационные артефакты.
- Закрыть или явно разобрать существующие `TODO`.
- Подготовить проект к более широкому использованию, а не только к локальному или внутреннему.

## Быстро

### 1. Привести в порядок CI и процесс релизов

Текущее состояние:

- CI workflow запускается на `push` в `develop` и на `pull_request` в `main`.
- Тесты запускаются только через `go test -v ./`, а не через `go test ./...`.
- Есть отдельный workflow для автотегов, но он не выглядит надёжной основой для релизного процесса.

Что сделать:

- Обновить `.github/workflows/go.yml`:
  - запуск на `push` в `main`
  - запуск на `pull_request`
  - использовать `go test ./...`
  - при необходимости добавить `go test -race ./...` для поддерживаемых окружений
- Добавить badge статуса сборки в `README.md`.
- Оставить CI только для проверки качества, без смешивания с релизной логикой.

Ожидаемый результат:

- По репозиторию сразу видно, что пакет регулярно проверяется.
- Базовые сигналы качества усиливаются почти без изменений в коде.

### 2. Подключить GoReleaser и публиковать релизы по push тега

Зафиксированное решение:

- Теги создаются и пушатся вручную.
- Автоматическое создание тегов через GitHub Actions не используется.
- `GoReleaser` запускается только на `push` версионного тега и создаёт GitHub Release.

Что сделать:

- Добавить `.goreleaser.yaml` в корень репозитория.
- Добавить `.github/workflows/release.yml`, который запускается на:
  - `push.tags: ['v*']`
- Workflow должен:
  - checkout репозитория с полной историей
  - setup Go
  - запускать `goreleaser release --clean`
  - публиковать GitHub Release и артефакты

Что убрать:

- Удалить текущий `tag.yml`, чтобы в репозитории не оставалось параллельной логики автотегов.

Операционный процесс:

- Версия определяется вручную.
- После подготовки изменений вручную создаётся тег вида:
  - `v2.5.0`
  - `v2.5.1`
- После push тега GitHub Actions публикует релиз автоматически.

Почему это правильный вариант:

- Нет скрытой логики bump-версии внутри workflow.
- Нет проблем с цепочкой workflow, когда один action пушит тег, а второй должен запускаться от этого события.
- Версионирование остаётся прозрачным и управляемым.

Ожидаемый результат:

- Релизы становятся видимыми и предсказуемыми.
- Процесс публикации остаётся простым: подготовил изменения, создал тег, запушил, получил релиз.

### 3. Закрыть технический долг по конкурентности

Текущее состояние:

- `Converter` использует изменяемое состояние через `pointerIndx`.
- Из-за этого в тестах закомментирован `t.Parallel()`.

Что сделать:

- Переработать `Converter` так, чтобы состояние одного вызова `Convert` было локальным.
- Убрать `pointerIndx` как общее изменяемое состояние из долгоживущей структуры.
- Включить обратно parallel tests в `converter_test.go`.
- Добавить тесты на race conditions.
- По возможности добавить `go test -race ./...` в CI.

Ожидаемый результат:

- Реализация станет безопаснее и понятнее.
- Будет проще объяснять реальные ограничения по goroutine safety.

### 4. Усилить валидацию конфига и обработку options

Текущее состояние:

- Валидация уже есть, но часть поведения остаётся неявной.
- Options сейчас реализованы через узкий `switch` с одним известным случаем.

Что сделать:

- Расширить `Modification.Validate()`:
  - проверка конфликтов separator'ов
  - более явная проверка template
  - по возможности проверка формы link и ссылок на теги
- Заменить ad hoc-обработку options на более явный механизм dispatch/registry.
- Явно описать поддерживаемые options в README.

Ожидаемый результат:

- Ошибки конфига будут ловиться раньше.
- Поведение системы станет предсказуемее при добавлении новых options.

### 5. Добрать недостающие поведенческие тесты

Что сделать:
